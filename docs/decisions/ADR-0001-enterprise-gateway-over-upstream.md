# ADR-0001: 企业网关覆盖上游的架构模式

> **状态：Superseded。** Gateway 兼容边界仍保留，但完整中期架构决策已由 [ADR-ENV-001](./ADR-ENV-001-MODULAR-MONOLITH-MIDTERM-ARCHITECTURE-2026-07.md) 取代。本文仅保留早期决策依据。

原状态：Accepted

日期：2026-06-11

---

## 背景

本项目基于上游 `hero8152/Infinite-Canvas`。上游项目持续维护 `main.py`、`static/`、`workflows/`、`API/`、`python/`、`VERSION` 等核心文件。

企业版需要在上游能力基础上增加多用户登录、权限隔离、画布归属、对话归属、审计日志和管理后台。如果直接在上游核心文件中维护企业功能，每次上游更新都会带来更高冲突风险和维护成本。

---

## 决策

企业能力优先通过 `enterprise gateway` 实现。

具体做法：

- 上游 Infinite Canvas 作为内部服务运行在 `127.0.0.1:3001`。
- 企业网关 `enterprise/gateway.py` 对外监听 `0.0.0.0:8000`。
- 企业网关负责认证、代理、拦截、过滤和用户上下文注入。
- 企业层使用独立数据库记录用户、归属和审计信息。
- 企业前端放在 `enterprise-static/`。
- 测试和诊断脚本放在 `enterprise/tests/`。
- 上游核心文件不作为企业功能主要开发入口。

---

## 后果

收益：

- 降低上游同步冲突。
- 企业功能边界清晰。
- 上游升级时更容易判断覆盖范围。
- Codex 长期维护时更不容易偏离方向。
- 可以把企业数据和上游数据分离维护。

代价：

- 企业网关需要承担代理和过滤复杂度。
- 某些上游前端行为难以通过代理层干预，可能需要向上游提交 bugfix。
- 每次上游更新后仍需执行诊断、冒烟和手工验证。

---

## 适用范围

本 ADR 适用于所有企业多用户能力开发，包括：

- 登录认证
- 用户管理
- 权限隔离
- 画布/对话归属
- 管理后台
- 审计日志
- 启动/停止与部署维护
- 上游同步后的验证流程

如果未来必须修改上游覆盖区域，应在对应 PR 中说明该修改为何无法通过企业层实现，并记录风险与回滚方式。
